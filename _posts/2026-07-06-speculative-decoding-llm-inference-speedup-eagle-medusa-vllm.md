# LLM 추론 속도를 2~6배 올리는 추측 디코딩: EAGLE-3, Medusa, vLLM 실전 가이드

LLM 서비스에서 가장 답답한 순간은 모델이 토큰을 한 개씩 천천히 내뱉을 때다. GPU가 충분히 빠름에도 불구하고, 모델은 이전 토큰의 결과를 기다렸다가 다음 토큰을 생성하는 autoregressive 방식 때문에 본질적으로 직렬 처리에 갇힌다. 양자화로 모델 크기를 줄이고, KV cache를 최적화해도 이 직렬성은 해결되지 않는다.

추측 디코딩(speculative decoding)은 이 직렬성을 공략하는 기술이다. 작은 모델로 여러 토큰을 "추측"하고, 큰 모델로 한 번에 "검증"하는 방식으로, 정확도 손실 없이 추론 속도를 2~6배 높인다. 2026년 현재 EAGLE-3, Medusa 등의 방법이 vLLM과 SGLang 같은 프로덕션 서빙 프레임워크에 통합되면서, 더 이상 실험실 기술이 아닌 실전 선택지가 됐다.

이 글에서는 추측 디코딩이 왜 작동하는지, EAGLE과 Medusa의 접근법이 어떻게 다른지, 그리고 실제 서빙 스택에 어떻게 적용하는지를 정리한다.

## 왜 autoregressive 디코딩이 병목인가

GPT, Llama, Mistral 같은 트랜스포머 기반 LLM은 모두 autoregressive 방식으로 동작한다. N번째 토큰을 생성하려면 1번부터 N-1번까지의 모든 토큰이 먼저 결정되어 있어야 한다. 매 스텝마다 모델의 전체 파라미터를 GPU의 HBM(고대역폭 메모리)에서 컴퓨팅 코어로 옮겨야 하고, 이 메모리 이동이 실제 연산보다 오래 걸린다.

이를 **메모리 바운드(memory-bound)** 상황이라고 부른다. GPU의 TFLOPS는 여유가 많지만, 메모리 대역폭이 병목이라서 연산 자원의 상당 부분이 유휴 상태로 낭비된다. 추측 디코딩은 바로 이 유휴 연산 자원을 활용한다 — 한 번의 forward pass에서 여러 토큰의 검증을 동시에 처리함으로써, 메모리 이동 횟수를 줄이는 것이다.

배치 크기가 작을 때(예: 단일 사용자 대화) 이 효과가 가장 크다. 배치가 커지면 GPU 연산이 이미 포화 상태에 가까워져서 추측 디코딩의 이득이 줄어든다. 이는 뒤에서 다시 다룬다.

## 추측 디코딩의 핵심 원리: 초안과 검증

추측 디코딩은 두 단계로 동작한다.

**1단계 — 초안 생성(Draft):** 작고 빠른 "드래프트 모델"이 몇 개의 다음 토큰을 빠르게 예측한다. 예를 들어 4개의 토큰을 추측한다고 하자.

**2단계 — 검증(Verify):** 큰 "타겟 모델"이 이 4개의 토큰을 한 번의 forward pass로 검증한다. 트랜스포머는 한 번에 N개 위치의 로짓을 계산할 수 있으므로, 4개 토큰을 개별적으로 4번 검증할 필요 없이 한 번에 처리한다.

여기서 핵심은 **검증 비용이 생성 비용과 거의 같다**는 점이다. 타겟 모델의 forward pass 한 번이면 4개 토큰의 정확성을 모두 판단할 수 있다. 드래프트 모델의 예측이 맞았으면 4개 토큰을 "공짜"로 얻은 셈이고, 틀린 부분부터는 타겟 모델의 정답으로 대체된다.

중요한 점은 **출력 분포가 보존된다**는 것이다. 거절-기반 재샘플링(rejection-based resampling)을 통해, 추측 디코딩의 최종 출력은 일반 디코딩과 수학적으로 동일한 확률 분포를 따른다. 즉, 품질 손실이 없다. 이것이 단순한 작은 모델 사용과 추측 디코딩의 근본적 차이다 — 작은 모델을 그냥 쓰면 품질이 떨어지지만, 추측 디코딩은 큰 모델의 품질을 유지하면서 속도만 높인다.

## 드래프트 모델의 세 가지 접근법

초안을 어떻게 생성하느냐가 각 방법론의 차이다. 2026년 현재 크게 세 가지 방향이 있다.

### 1. 별도 드래프트 모델 (전통적 방식)

Llama-70B를 타겟으로 쓰면서 Llama-7B나 1B를 드래프트로 쓰는 방식이다. 개념이 단순하고 구현이 쉽지만, 드래프트 모델과 타겟 모델의 토크나이저·아키텍처가 호환되어야 한다. vLLM은 `--speculative-model` 플래그로 이 방식을 지원한다.

```bash
# vLLM에서 별도 드래프트 모델 사용
vllm serve meta-llama/Meta-Llama-3-70B \
  --speculative-model meta-llama/Meta-Llama-3-8B \
  --num-speculative-tokens 5
```

한계는 드래프트 모델의 정확도가 낮을 때 수락률(acceptance rate)이 떨어져 속도 향상이 사라진다는 점이다. 또한 별도 모델의 가중치를 메모리에 추가로 올려야 하므로 VRAM 비용이 증가한다.

### 2. Medusa — 다중 헤드 자기 추측

Medusa(arXiv:2401.10774)는 별도 드래프트 모델 대신, 타겟 모델 자체에 추가 디코딩 헤드를 붙이는 방식이다. 메인 헤드가 다음 토큰을 예측하듯, 추가 헤드들은 2번째, 3번째, 4번째 미래 토큰을 동시에 예측한다.

이 방식의 장점은 별도 모델이 필요 없다는 것이다. 헤드만 추가 학습하면 되므로 메모리 오버헤드가 적다. Medusa는 트리 기반 어텐션(tree-based attention) 메커니즘으로 여러 후속 토큰 조합을 동시에 검증한다.

Medusa에는 두 가지 학습 모드가 있다. Medusa-1은 타겟 모델을 고정(freeze)하고 헤드만 학습해 무손실 가속을 달성하고, Medusa-2는 타겟 모델과 함께 헤드를 파인튜닝해 더 높은 정확도와 속도를 얻는 대신 특수한 학습 레시피가 필요하다.

### 3. EAGLE — 특징 수준 자기 추측

EAGLE(arXiv:2401.15077, ICML 2024)은 메인 스트림의 발전 방향이 된 방법이다. EAGLE은 타겟 모델의 최상위 특징(feature)을 재사용하여, 토큰이 아닌 특징 수준에서 autoregression을 수행한다. 이렇게 하면 드래프트 모델이 타겟 모델의 "생각 과정"을 직접 이어받을 수 있어 정확도가 크게 높아진다.

**EAGLE-3**(arXiv:2503.01840)은 이 접근의 최신 발전이다. EAGLE-1/2가 특징 예측에 의존했던 것과 달리, EAGLE-3은 특징 예측을 버리고 직접 토큰 예측으로 전환하면서, 다중 계층 특징 융합(multi-layer feature fusion)을 도입했다. 이를 통해 드래프트 모델이 학습 데이터 확장의 이점을 온전히 누릴 수 있게 됐다.

EAGLE-3 논문에 따르면 최대 **6.5배의 속도 향상**을 달성했으며, 이는 EAGLE-2 대비 약 1.4배의 추가 개선이다. SGLang 프레임워크 기준 배치 크기 64에서 처리량(throughput) 기준 1.38배 향상을 기록했다.

> **참고**: EAGLE-3의 6.5x 수치는 논문의 최적 조건에서의 결과다. 실제 프로덕션 환경에서는 워크로드, 배치 크기, 모델 구조에 따라 2~4배 수준이 일반적이다. 배치가 크거나(64+) 긴 컨텍스트(128K+)를 처리할 때는 이득이 줄어든다.

### 접근법 비교 요약

| 방식 | 드래프트 | 추가 메모리 | 속도 향상 | 정확도 |
|------|---------|------------|----------|--------|
| 별도 모델 | 독립 소모델 | 높음 (별도 가중치) | 2~3x | 무손실 |
| Medusa | 타겟 모델 헤드 | 낮음 (헤드만) | 2~3x | 무손실(Medusa-1) |
| EAGLE-3 | 특징 기반 자기 추측 | 중간 | 3~6x | 무손실 |

## vLLM과 SGLang에서 실전 적용

2026년 현재, 두 주요 오픈소스 서빙 프레임워크가 추측 디코딩을 기본 지원한다.

### vLLM

vLLM의 공식 문서에 따르면, 세 가지 추측 디코딩 방식을 지원한다.

```bash
# 방법 1: n-gram 기반 (드래프트 모델 불필요, 메모리 추가 비용 없음)
vllm serve meta-llama/Meta-Llama-3-8B \
  --speculative-model "[ngram]" \
  --num-speculative-tokens 5 \
  --ngram-prompt-path-max-length 1024

# 방법 2: 별도 드래프트 모델
vllm serve meta-llama/Meta-Llama-3-70B \
  --speculative-model meta-llama/Meta-Llama-3-8B \
  --num-speculative-tokens 5

# 방법 3: EAGLE/Medusa 헤드가 통합된 모델
vllm serve yuhuili/EAGLE-LLaMA3-Instruct-8B \
  --speculative-model yuhuili/EAGLE-LLaMA3-Instruct-8B \
  --speculative-draft-tensor-parallel-size 1
```

n-gram 방식은 드래프트 모델이 아예 필요 없다. 프롬프트 내에서 반복되는 패턴을 찾아 후보 토큰을 제안하는 방식으로, 코드 생성이나 긴 문서 요약처럼 반복성이 높은 워크로드에서 효과적이다. 메모리 추가 비용이 없다는 것이 큰 장점이다.

### SGLang

SGLang은 EAGLE-3 통합에 적극적이다. EAGLE-3 논문의 공식 벤치마크가 SGLang 프레임워크에서 수행됐을 정도로 긴밀하게 협력하고 있다. SGLang은 RadixAttention 기반의 prefix caching과 추측 디코딩을 결합할 수 있어, 에이전트 워크플로우처럼 반복되는 시스템 프롬프트가 많은 환경에서 시너지 효과를 낸다.

### 구조화된 출력과의 상호작용

추측 디코딩을 쓸 때 자주 묻는 질문이 있다. "JSON 스키마 강제나 정규식 제약을 걸면 속도 향상이 유지되나?"

답은 "부분적으로"다. 구조화된 출력 제약(constrained decoding)은 각 위치에서 선택 가능한 토큰의 범위를 좁힌다. 이는 드래프트 모델이 더 정확한 예측을 할 수 있게 해서 수락률을 높일 수도 있지만, 반대로 제약 조건 자체가 연산 오버헤드를 추가해서 순이득을 줄일 수도 있다. vLLM과 SGLang 모두 구조화 출력 + 추측 디코딩 조합을 지원하지만, 실제 성능은 스키마 복잡도와 워크로드에 따라 달라지므로 자체 벤치마크가 필요하다.

## 언제 추측 디코딩이 효과적인가, 아닌가

추측 디코딩이 항상 유리한 것은 아니다. 적용 전 다음 조건을 확인해야 한다.

**효과적인 조건:**
- **낮은 배치 크기(1~8):** GPU 연산이 포화되지 않은 상태에서 추측의 이점이 가장 크다. 실시간 채팅, 1:1 에이전트 대화 등이 여기에 해당한다.
- **짧은~중간 컨텍스트(4K~32K):** KV cache가 메모리를 크게 잡아먹지 않아 드래프트 모델을 올릴 여유가 있다.
- **반복성 높은 출력:** 코드 생성, 데이터 포맷 변환, 긴 문서 요약 등은 n-gram 방식의 수락률이 높다.
- **지연 시간(latency)이 핵심 메트릭인 경우:** 처리량(throughput)보다 토큰당 응답 속도가 중요한 서비스.

**효과가 떨어지는 조건:**
- **높은 배치 크기(64+):** 이미 GPU가 연산으로 포화되어 추측 검증의 "공짜" 이점이 사라진다. 이런 환경에서는 추측 디코딩이 오히려 처리량을 떨어뜨릴 수 있다.
- **매우 긴 컨텍스트(128K+):** 긴 컨텍스트에서는 어텐션 연산 비용이 높아져 검증 단계의 오버헤드가 커진다. 또한 KV cache 메모리가 부족해 드래프트 모델을 올릴 공간이 없을 수 있다.
- **창의적/발산적 생성:** 드래프트 모델이 타겟 모델의 다양한 출력을 예측하기 어려워 수락률이 떨어진다.

이런 이유로 추측 디코딩을 프로덕션에 도입할 때는 A/B 테스트가 필수다. 특정 워크로드에서는 켜고, 다른 워크로드에서는 끄는 동적 제어가 필요할 수 있다.

## 핵심 요약 및 다음 단계

- **autoregressive 병목의 본질**은 연산이 아니라 메모리 이동이다. 추측 디코딩은 한 번의 forward pass로 여러 토큰을 검증해 이 병목을 우회한다.
- **출력 품질은 보존된다.** 거절 기반 재샘플링을 통해 타겟 모델의 원래 분포와 수학적으로 동일한 결과를 낸다. 단순히 작은 모델로 대체하는 것과 근본적으로 다르다.
- **EAGLE-3가 2026년의 사실상 표준**이다. 특징 수준 자기 추측 + 다중 계층 융합으로 최대 6.5배 속도 향상을 달성했으며, vLLM과 SGLang에 통합되어 있다.
- **배치가 클 때는 주의가 필요하다.** 추측 디코딩의 이점은 GPU가 유휴 상태일 때 가장 크다. 고배치 환경에서는 처리량이 오히려 떨어질 수 있다.
- **가장 먼저 시도할 것:** n-gram 기반 추측 디코딩이다. 드래프트 모델이 필요 없고 메모리 추가 비용이 없어서, 기존 vLLM 배포에 플래그 하나만 추가해 효과를 볼 수 있다.

추측 디코딩은 LLM 추론 최적화의 한 축이다. KV cache 최적화, 양자화, continuous batching과 조합했을 때 시너지가 발생한다. 다음 단계로는 자체 워크로드에서 EAGLE-3 헤드를 학습하거나, SGLang의 RadixAttention과 결합한 에이전트 시나리오 벤치마크를 시도해 볼 수 있다.
