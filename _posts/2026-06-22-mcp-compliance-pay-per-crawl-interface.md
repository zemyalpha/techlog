# AI 에이전트가 컴플라이언스와 과금을 읽는 시대: MCP와 pay-per-crawl

AI 에이전트 이야기를 할 때 우리는 종종 “무엇을 더 잘 생성하느냐”에만 집중합니다. 그런데 최근 흐름을 보면 질문이 달라지고 있습니다. **무엇을 읽게 할 것인가, 무엇을 쓰게 할 것인가, 그리고 그 행동에 어떻게 책임과 비용을 붙일 것인가**가 더 중요해지고 있습니다.

이 변화는 서로 다른 두 발표에서 선명하게 드러납니다. 하나는 Sumsub의 MCP 통합입니다. Sumsub는 AI 에이전트가 AML 정책 문서를 바탕으로 컴플라이언스 워크플로를 자동 구성할 수 있다고 밝혔습니다. 다른 하나는 Cloudflare의 pay-per-crawl입니다. Cloudflare는 콘텐츠 소유자가 AI 크롤러의 접근을 통제하고, 경우에 따라 접근료를 청구할 수 있는 방식을 내놨습니다. 한쪽은 **운영 절차의 에이전트화**, 다른 한쪽은 **접근 권한의 가격화**입니다.

## 1) 왜 지금 이 조합이 중요할까

Anthropic은 MCP를 AI 어시스턴트를 데이터가 있는 시스템에 연결하는 개방형 표준으로 설명합니다. 이 말은 단순합니다. 모델이 똑똑해지는 것만으로는 부족하고, 실제 업무 시스템과 안전하게 연결되어야 한다는 뜻입니다. 그래서 MCP는 “채팅창 옆의 장난감”이 아니라 **실제 업무 실행 계층**으로 읽어야 합니다.

Sumsub 사례가 흥미로운 이유도 여기에 있습니다. 컴플라이언스 설정은 원래 사람이 정책 문서를 읽고, 규칙을 해석하고, 시스템 설정을 맞추는 반복 작업입니다. 이 절차가 에이전트의 입력/출력 형태로 바뀌면, 문서는 더 이상 저장용 PDF가 아니라 **실행 가능한 정책**이 됩니다. 즉, 정책이 곧 인터페이스가 됩니다.

Cloudflare의 pay-per-crawl도 같은 축에 있습니다. 웹의 공개 콘텐츠는 오랫동안 “가져가도 되는 것”처럼 취급돼 왔지만, AI 시대에는 그 가정이 흔들립니다. 접근을 허용할지, 차단할지, 아니면 비용을 붙일지 결정해야 하기 때문입니다. 결국 웹사이트 운영자는 더 이상 단순한 퍼블리셔가 아니라 **접근 정책의 운영자**가 됩니다.

## 2) 실전 구현 관점: 에이전트에게는 ‘넓은 권한’보다 ‘좁은 도구’가 낫다

이 흐름을 실제 시스템에 적용할 때 핵심은 하나입니다. 에이전트에게 관리자 권한을 주는 것이 아니라, **작은 작업 단위로 분해된 도구**를 주는 것입니다.

예를 들어 컴플라이언스 자동화는 다음처럼 나눌 수 있습니다.

```json
{
  "name": "configure_compliance_workspace",
  "description": "정책 문서를 해석해 검토용 워크스페이스를 구성한다",
  "inputSchema": {
    "type": "object",
    "properties": {
      "policy_document": {"type": "string"},
      "jurisdiction": {"type": "string"},
      "risk_profile": {"type": "string", "enum": ["low", "medium", "high"]},
      "human_approval_required": {"type": "boolean"}
    },
    "required": ["policy_document", "jurisdiction"]
  }
}
```

이렇게 하면 에이전트는 “무엇을 할지” 제안할 수는 있어도, 마지막 단계는 사람이 승인하도록 설계할 수 있습니다. 이 구조가 중요한 이유는 규정 준수, 보안, 감사 추적이 모두 **행동의 흔적**을 요구하기 때문입니다.

접근 과금도 비슷합니다. 콘텐츠를 전부 막거나 전부 여는 대신, 정책을 분리해야 합니다.

```yaml
crawler_policy:
  allow:
    - verified_search_bot
    - partner_indexer
  deny:
    - unknown_ai_crawler
    - bulk_scraper
  pricing:
    default_currency: USD
    model: pay_per_request
    notes: "접근 허용 여부와 과금 규칙을 분리해서 운영"
  audit:
    log_requests: true
    retain_days: 90
```

핵심은 기술보다 운영입니다. **승인, 로깅, 롤백, 감사**가 함께 있어야 에이전트가 실제 업무에 들어갈 수 있습니다.

## 3) 이 흐름이 보여주는 시장 변화

이 두 발표를 같이 보면 시장의 초점이 바뀌고 있음을 알 수 있습니다.

- 예전: 모델 성능, 응답 길이, 벤치마크 점수
- 지금: 권한, 정책, 비용, 감사 가능성
- 다음: 에이전트가 시스템을 “읽고 쓰는” 범위를 누가 통제하느냐

즉, 경쟁 포인트는 더 이상 “누가 더 말을 잘하나”가 아닙니다. **누가 더 안전하게 시스템을 바꾸게 만들 수 있나**입니다.

이 관점에서 보면 Sumsub는 컴플라이언스를 실행 가능한 업무로 바꾸고 있고, Cloudflare는 콘텐츠 접근을 경제적 계약으로 바꾸고 있습니다. 둘 다 공통적으로 “AI가 실제 시스템을 만질 때 필요한 중간 계층”을 만들고 있습니다.

## 4) 주의할 점과 실전 팁

1. **에이전트에게 원본 관리자 API를 바로 주지 마십시오.**
   먼저 읽기 전용 조회와 제안 단계를 분리하는 편이 안전합니다.
2. **정책 문서는 사람용에서 기계 실행용으로 재구성해야 합니다.**
   모호한 문장보다, 조건·예외·승인 절차가 명시된 문서가 좋습니다.
3. **과금은 차단의 대체재가 아닙니다.**
   허용/거부/과금/로그를 각각 독립적으로 설계해야 합니다.
4. **감사 로그가 없으면 자동화가 아니라 방치입니다.**
   누가 언제 무엇을 바꿨는지 남지 않으면 운영이 무너집니다.

## 결론: AI 에이전트의 다음 전장은 정책 인터페이스다

- MCP는 AI를 외부 시스템에 연결하는 개방형 표준으로 읽어야 합니다.
- Sumsub 사례는 컴플라이언스가 문서에서 실행으로 옮겨가고 있음을 보여줍니다.
- Cloudflare의 pay-per-crawl은 접근 권한이 곧 수익 모델이 될 수 있음을 보여줍니다.
- 앞으로 중요한 것은 모델 자체보다 **권한·과금·감사 레이어**입니다.
- 가장 먼저 할 일은 “에이전트가 무엇을 할 수 있나?”보다 “에이전트가 무엇을 읽고, 무엇을 쓰고, 누가 승인하나?”를 정의하는 것입니다.

지금 AI를 잘 쓰는 팀은 더 똑똑한 모델을 고르는 팀이 아니라, **정책을 기계가 읽을 수 있게 바꾸는 팀**일 가능성이 큽니다. 그게 바로 오늘의 실전 경쟁력입니다.
