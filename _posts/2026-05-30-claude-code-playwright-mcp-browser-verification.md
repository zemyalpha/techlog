# Claude Code와 Playwright MCP를 같이 쓰는 이유: 코드 수정과 브라우저 검증 분리하기

요즘의 코딩 에이전트는 코드를 잘 쓰는 것만으로는 부족합니다. 수정한 코드가 실제 브라우저에서 어떻게 보이는지, 클릭과 라우팅이 기대대로 동작하는지, 에러가 화면 뒤에서 숨어 있지는 않은지까지 확인해야 합니다. 그래서 실무에서는 **코드 수정 레이어**와 **브라우저 검증 레이어**를 분리하는 편이 훨씬 안정적입니다.

이 글의 핵심은 단순합니다. **Claude Code는 코드베이스를 읽고, 여러 파일을 고치고, 테스트를 돌리는 역할**에 강하고, **Playwright MCP는 브라우저 안에서 페이지 구조를 확인하는 역할**에 강합니다. 여기에 **MCP Registry**를 붙이면 필요한 서버를 찾고 관리하는 출발점까지 갖출 수 있습니다.

## 1) 왜 둘을 나눠야 하나

Anthropic은 Claude Code를 “agentic coding system”이라고 설명합니다. 공식 설명에 따르면 Claude Code는 코드베이스를 읽고, 파일 전반을 수정하고, 테스트를 실행하며, 커밋된 코드를 전달하는 방식으로 동작합니다. 기본값도 보수적이라서, 파일을 바꾸거나 명령을 실행하기 전에 먼저 확인을 요구합니다. 즉, Claude Code는 **변경과 재현성**에 강한 쪽입니다.

반면 Playwright MCP는 다릅니다. 공식 README는 이 서버를 **Playwright 기반의 MCP 서버**라고 소개하고, LLM이 웹 페이지와 상호작용할 때 **스크린샷 대신 structured accessibility snapshots**를 사용한다고 설명합니다. 이 방식은 화면을 “보는” 것보다 구조를 “읽는” 데 가깝습니다. 그래서 브라우저 검증, 탐색적 자동화, 자기수정형 테스트 같은 작업에 잘 맞습니다.

중요한 포인트는, 모든 브라우저 작업을 MCP로 몰아넣는 것이 정답은 아니라는 점입니다. Playwright MCP 저장소는 코딩 에이전트라면 오히려 **CLI + SKILLS**가 더 토큰 효율적일 수 있다고도 말합니다. 큰 스키마나 긴 accessibility tree를 매번 컨텍스트에 싣지 않아도 되기 때문입니다. 즉, **코드 수술은 Claude Code**, **브라우저 확인은 필요한 순간에만 Playwright MCP**라는 분리가 현실적입니다.

## 2) 실전 구성: 설치와 연결은 이렇게 단순하게 시작한다

가장 먼저 할 일은 Playwright MCP를 에이전트가 접근할 수 있게 연결하는 것입니다. 공식 README가 제시하는 표준 설정은 아래와 같습니다.

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest"
      ]
    }
  }
}
```

Claude Code를 쓰는 경우에는 저장소 README에 나온 것처럼 CLI로 서버를 추가할 수 있습니다.

```bash
claude mcp add playwright npx @playwright/mcp@latest
```

그다음 흐름은 이렇게 잡으면 좋습니다.

1. Claude Code로 기능 수정이나 리팩터링을 먼저 수행한다.
2. 단위 테스트와 린트를 돌려서 코드 레벨의 오류를 먼저 걷어낸다.
3. UI가 영향을 받는 변경이라면 Playwright MCP로 브라우저를 열어 확인한다.
4. 버튼, 라우팅, 접근성 상태, 렌더링 오류를 읽고 최소 수정만 반영한다.
5. 다시 테스트와 브라우저 검증을 반복한다.

이 흐름의 장점은, 브라우저 검증이 “무작정 전체 자동화”가 아니라 **변경이 실제로 사용자 경험을 깨는지 확인하는 최종 관문**이 된다는 점입니다.

## 3) MCP Registry는 어디에 끼워 넣어야 하나

MCP Registry의 역할은 의외로 소박하지만 중요합니다. 공식 페이지는 자신을 **Official MCP Registry**라고 부르며, 목적을 **Discover Model Context Protocol servers**라고 적고 있습니다. 또 production, staging, local, custom 같은 API Base URL 옵션도 보여줍니다.

실무 관점에서는 이게 꽤 유용합니다. 새 서버를 찾을 때마다 비공식 문서를 뒤지는 대신, 먼저 Registry에서 존재 여부와 위치를 확인할 수 있기 때문입니다. 에이전트 툴링이 늘어날수록 “무엇을 설치할지”보다 “무엇을 공식적으로 신뢰할지”가 더 중요해지는데, Registry는 그 첫 관문 역할을 합니다.

제가 추천하는 순서는 다음과 같습니다.

- **Registry에서 서버를 찾는다.**
- **필요한 서버만 연결한다.**
- **브라우저 검증이 필요한 경우에만 Playwright MCP를 켠다.**
- **코드 변경은 계속 Claude Code가 담당하게 둔다.**

이렇게 하면 툴이 늘어나도 컨텍스트가 산만해지지 않습니다.

## 4) 팀이 자주 하는 실수와 더 나은 패턴

가장 흔한 실수는 “에이전트가 다 할 수 있으니 하나의 도구로 끝내자”라고 생각하는 것입니다. 하지만 실제로는 반대입니다. 하나의 도구에 너무 많은 책임을 넣으면 디버깅이 어려워지고, 실패 원인이 코드인지 브라우저인지 툴 설정인지 분리되지 않습니다.

더 나은 패턴은 이렇습니다.

- **코드 변경 책임**은 Claude Code에 둔다.
- **브라우저 사실 확인**은 Playwright MCP에 둔다.
- **툴 발견과 관리**는 MCP Registry에 둔다.
- **최종 승인**은 사람이나 CI 단계가 맡는다.

이 구조는 느려 보일 수 있지만, 실제로는 재작업을 줄여서 더 빠릅니다. 화면이 깨졌을 때 코드부터 다시 뜯는 대신, 브라우저 검증에서 먼저 문제가 드러나기 때문입니다.

## 결론: 에이전트를 한 덩어리로 보지 말고, 역할별로 쪼개자

오늘의 실전 요약은 간단합니다.

- Claude Code는 코드베이스 수정과 테스트에 강하다.
- Playwright MCP는 브라우저 검증과 페이지 구조 확인에 강하다.
- MCP Registry는 서버를 발견하고 관리하는 출발점이다.
- 모든 브라우저 작업을 MCP로 몰지 말고, 필요한 순간에만 켜는 편이 효율적이다.
- 코딩 에이전트의 품질은 “얼마나 많이 자동화했는가”보다 “어디서 검증을 멈췄는가”로 갈린다.

당장 해볼 첫 단계는 하나입니다. 지금 관리 중인 웹 기능 하나를 골라서, **코드 수정용 에이전트**와 **브라우저 검증용 에이전트**의 책임을 분리해 보세요. 그 순간부터 자동화는 더 조용해지고, 디버깅은 더 빨라집니다.
