# MCP Registry·Chronicle·Playwright MCP로 만드는 디버그 가능한 에이전트 스택

에이전트를 잘 만든다는 말은 이제 "좋은 답을 만든다"가 아니다. 실제 운영에서는 도구를 어떻게 찾고, 작업 내역을 어떻게 남기고, 브라우저 동작을 어떻게 다시 검증하느냐가 더 중요하다. 최근 공개된 GitHub Chronicle, MCP Registry, Playwright MCP를 함께 보면 이 방향이 꽤 선명해진다. 하나는 도구를 찾게 해주고, 하나는 세션을 다시 읽게 해주고, 하나는 화면 대신 페이지 상태로 검증하게 해준다.

이 조합이 좋은 이유는 단순하다. 에이전트 실패는 대부분 "모델이 못해서"가 아니라 "어디서 무엇을 했는지 복구할 수 없어서" 생긴다. 따라서 운영 기준도 바뀌어야 한다. 발견 가능한 도구, 추적 가능한 세션, 재현 가능한 검증이 있어야 한다.

## 1) 먼저, 도구는 레지스트리에서 발견돼야 한다

MCP Registry의 핵심은 간단하다. MCP 클라이언트가 쓸 수 있는 서버 목록을 제공해, 도구를 일종의 카탈로그처럼 다룰 수 있게 한다. README는 이를 "MCP 서버용 앱 스토어"에 비유한다. 이 비유가 중요한 이유는, 에이전트가 커질수록 도구를 하드코딩한 설정 파일보다 검색 가능한 목록이 더 안전하고 관리하기 쉬워지기 때문이다.

실무적으로는 이렇게 생각하면 된다.

- 새 프로젝트마다 도구를 복붙하지 말고
- 공통적으로 쓸 서버는 먼저 레지스트리에서 찾고
- 정말 필요한 것만 로컬 설정에 고정한다

이렇게 하면 팀마다 같은 브라우저 자동화 도구를 다른 이름으로 중복 등록하는 일을 줄일 수 있다.

## 2) 기억은 로그가 아니라 세션이어야 한다

GitHub Chronicle은 이 지점을 정확히 찌른다. GitHub Changelog는 `/chronicle`가 각 Copilot 세션의 히스토리를 바탕으로 standup summary, personalized tips, custom instructions를 만든다고 설명한다. GitHub Docs 쪽 설명도 꽤 구체적이다. Copilot CLI를 쓰면 세션 데이터가 로컬에 기록되고, 여기에는 프롬프트, Copilot 응답, 사용한 도구, 수정된 파일 정보가 포함된다. 기본적으로 이 기록은 GitHub 계정과도 동기화된다.

즉, Chronicle이 주는 가치는 단순한 저장이 아니다. "무슨 생각으로 이 작업을 했는가"를 다시 읽을 수 있게 만드는 것이다. 이게 없으면 에이전트는 매번 새로 시작한다. 반대로 세션이 쌓이면, 다음 작업은 이전 작업의 재시작이 된다.

에이전트 운영에서 이건 꽤 큰 차이다.

- 로그만 있으면: 실패를 보관한다
- 세션이 있으면: 실패의 맥락을 되돌린다
- 세션 요약이 있으면: 다음 작업의 입력으로 재사용한다

## 3) 검증은 스크린샷보다 페이지 상태가 낫다

Playwright MCP는 LLM이 웹 페이지를 브라우저 자동화 방식으로 다룰 수 있게 해주는 MCP 서버다. README는 이 서버가 Playwright를 이용해 browser automation capabilities를 제공하고, structured accessibility snapshots를 사용해 스크린샷 중심 접근을 피할 수 있다고 설명한다. 이 점이 중요하다. 화면을 이미지로 읽는 방식은 보기에는 직관적이지만, 에이전트 디버깅에는 페이지 구조가 더 유리하다.

설정도 복잡하지 않다. README에 있는 표준 구성은 아래와 같다.

```js
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest"
      ]
    }
  }
}
```

VS Code에서는 다음처럼 바로 추가할 수도 있다.

```bash
code --add-mcp '{"name":"playwright","command":"npx","args":["@playwright/mcp@latest"]}'
```

이런 식으로 붙여두면 에이전트는 단순히 "열어봤다"가 아니라, 어떤 페이지 상태였고 어떤 요소가 보였는지까지 남길 수 있다.

## 4) 실제 운영은 3단으로 보면 된다

내가 추천하는 최소 스택은 이렇다.

```yaml
agent_stack:
  discover: "MCP Registry에서 서버 찾기"
  remember: "/chronicle로 세션 요약 읽기"
  verify: "Playwright MCP로 브라우저 상태 확인"
```

이 구조가 주는 장점은 명확하다.

1. **발견**: 새 도구를 찾을 때 레지스트리를 먼저 본다.
2. **기억**: 작업이 끝나면 세션을 요약해 다음 입력으로 쓴다.
3. **검증**: UI 변경이나 폼 제출은 Playwright MCP로 다시 확인한다.

이 세 단계가 분리되면, 에이전트가 커져도 디버깅 비용이 폭증하지 않는다. 반대로 세 단계를 한 덩어리로 섞어버리면, "왜 실패했는지"를 다시 재구성하는 데 시간이 다 간다.

## 결론: 에이전트의 품질은 답변보다 복구 가능성에 달려 있다

- MCP Registry는 도구를 숨겨진 설정이 아니라 발견 가능한 인프라로 바꾼다.
- Chronicle은 세션을 일회성 기록이 아니라 재사용 가능한 맥락으로 만든다.
- Playwright MCP는 화면 감상이 아니라 구조화된 검증을 가능하게 한다.
- 세 가지를 같이 쓰면 에이전트는 더 화려해지는 게 아니라 더 디버그 가능해진다.

오늘 바로 해볼 일은 하나다. 지금 쓰는 에이전트 워크플로우에서 "도구 발견", "세션 요약", "브라우저 검증"이 각각 어디에 있는지 한 줄씩 적어보자. 그 세 줄이 보이면, 이미 절반은 운영 가능한 상태다.
