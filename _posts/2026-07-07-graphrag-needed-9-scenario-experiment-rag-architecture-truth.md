# GraphRAG가 정말 필요한가: 9개 시나리오 실험이 말해주는 RAG 선택의 진실

2026년, RAG(검색 증강 생성) 담론의 중심에는 GraphRAG와 Agentic RAG가 있습니다. 컨퍼런스 발표, 기술 블로그, 프로젝트 제안서 어디서나 "우리도 GraphRAG 도입해야 하는 거 아닌가요?"라는 질문이 따라옵니다. 합리적인 질문입니다. 지식 그래프로 엔티티 관계를 탐색하고, 에이전트가 다단계로 검색을 반복한다면 단순 벡터 검색보다 분명히 나을 것처럼 보이니까요.

하지만 2026년 6월 24일 arXiv에 발표된 한 논문이 이 직관에 찬물을 끼얹습니다. AWS Generative AI Innovation Center와 Cisco 연구진이 작성한 *"Is GraphRAG Needed? From Basic RAG to Graph-/Agentic Solutions with Context Optimization"* (arXiv:2606.25656)는 9개의 RAG 시나리오를 정밀 의학 데이터셋에서 체계적으로 비교했습니다. 결과는 예상과 달랐습니다 — **문서에 1-hop 관계만 덧붙인 가장 단순한 RAG가, 복잡한 GraphRAG와 Agentic RAG 구현의 상당수를 Hit@1 정확도에서 능가**했습니다.

이 글은 그 실험 데이터를 중심으로, GraphRAG와 Agentic RAG가 실제로 빛을 발하는 조건과 그렇지 않은 조건을 구분하는 실전 가이드를 제공합니다.

## 왜 이 실험 결과에 주목해야 하는가

그동안 GraphRAG vs 일반 RAG 비교는 개념 설명이나 단편적 벤치마크가 대부분이었습니다. "관계형 질문에 강하다", "인덱싱 비용이 비싸다"는 정성적 평가는 많았지만, 동일한 데이터셋·동일한 LLM·동일한 평가 메트릭 하에서 9단계의 아키텍처 복잡도를 체계적으로 비교한 연구는 드물었습니다.

이 논문이 특별한 이유는 세 가지입니다.

첫째, **평가 메트릭이 검색이 아니라 생성 품질**입니다. 기존 RAG 벤치마크는 "정답 문서가 검색 결과 상위 K개 안에 있는가"(Recall@K)를 측정하는 경우가 많습니다. 하지만 이 논문은 LLM이 실제로 생성한 답변에서 추출한 엔티티 ID가 정답과 일치하는지를 측정합니다(Hit@1, Hit@5, MRR). 검색이 잘 됐다고 답변까지 좋은 것은 아니라는 점을 이 메트릭 설계가 전제하고 있습니다.

둘째, **반정형화(semi-structured) 지식 베이스**를 다룹니다. 단순 텍스트 코퍼스가 아니라, 텍스트 문서와 명시적 관계 그래프가 공존하는 환경입니다. 이것이 바로 GraphRAG가 이론적으로 강점을 발휘해야 하는 조건입니다.

셋째, **현실적인 제약을 그대로 반영**합니다. LLM 컨텍스트 한계, 토큰 비용, 멀티턴 에이전트 세션에서의 컨텍스트 누적 문제 등 프로덕션에서 마주치는 문제를 실험 설계에 넣었습니다.

## 9개 시나리오: 복잡도의 사다리

논문은 RAG 아키텍처를 세 그룹으로 나누고, 각각을 단계적으로 복잡하게 만듭니다.

**Regular RAG (시나리오 1-2)**

- 시나리오 1: 엔티티 설명 문서만 인덱싱하는 가장 기본적인 벡터 RAG
- 시나리오 2: 각 문서에 1-hop 이웃 관계를 텍스트로 덧붙여 인덱싱

**GraphRAG (시나리오 3-5)**

- 시나리오 3: 미리 정의된 지식 그래프만 사용 (그래프 순회)
- 시나리오 4: 문서에서 자동으로 계산된 KG + 벡터 검색
- 시나리오 5: 벡터 검색 + 미리 정의된 KG의 하이브리드

**Modular & Agentic RAG (시나리오 6-9)**

- 시나리오 6: 쿼리 재작성 → 검색 → 재랭킹의 고정 파이프라인
- 시나리오 7: 모듈들을 에이전트의 도구로 제공 (ReAct 루프)
- 시나리오 8: 최소 도구만 주고 자율 추론에 맡김
- 시나리오 9: 시나리오 8에 KG 검색 도구 추가

실험 환경을 요약하면: STaRK-Prime 정밀 의학 데이터셋(약 129K 엔티티, 원본 PrimeKG의 관계를 확장해 구성), LLM은 Claude 3.7 Sonnet(Anthropic, 2025년 2월 출시), 그래프 저장소는 Amazon Neptune, 벡터 저장소는 OpenSearch Serverless입니다. PrimeKG 자체는 Harvard Zitnik Lab이 Nature Scientific Data에 발표한 정밀 의학 지식 그래프로 129,375 노드와 약 405만 관계를 가집니다.

## 결과: 단순함이 이긴 곳

핵심 결과를 Hit@1(첫 번째 답이 정답인 비율) 기준으로 정리합니다.

| 시나리오 | Hit@1 | 특징 |
|---|---|---|
| 시나리오 2 (문서+1-hop 관계) | **0.6972** | 가장 단순한 강화 RAG |
| 시나리오 8 (자율 에이전트, 최소 도구) | 0.6881 | Agentic RAG 최고 성적 |
| 시나리오 7 (에이전트+모듈 도구) | 0.6514 | Agentic RAG |
| 시나리오 5 (벡터+KG 하이브리드, 최적 설정) | 0.6514 | GraphRAG 최고 성적 |
| 시나리오 1 (순수 벡터) | 0.6147 | 기본선 |
| 시나리오 4 (계산된 KG+벡터) | 0.5596 | GraphRAG |
| 시나리오 3 (KG만) | **0.1376** | 그래프만 쓰면 거의 작동 안 함 |

가장 놀라운 발견을 세 가지로 요약할 수 있습니다.

**첫째, 시나리오 2의 부상입니다.** 문서에 1-hop 관계를 텍스트로 직렬화해 넣기만 했을 뿐인데, 실시간 그래프 순회나 다단계 에이전트 추론 없이도 관계형 컨텍스트의 이점을 LLM에 전달합니다. Hit@1 0.6972로, 가장 정교한 GraphRAG 하이브리드(0.6514)와 자율 에이전트(0.6881)를 모두 제쳤습니다.

**둘째, 그래프만 쓰면(시나리오 3) 거의 작동하지 않습니다.** Hit@1이 0.1376에 불과합니다. 텍스트 문서 없이 순수 그래프 삼중항(triplet)만으로는 LLM이 답을 생성하기 어렵습니다. GraphRAG가 강하려면 벡터 검색과 결합해야 한다는 것을 보여줍니다.

**셋째, 복잡도가 성능 비례로 올라가지 않습니다.** 시나리오 1→2로 갈 때 0.6147→0.6972로 확 뜁니다(관계 정보 추가의 효과). 하지만 시나리오 2→5(GraphRAG 하이브리드)로 갈 때는 오히려 0.6972→0.6514로 떨어집니다. 더 정교한 검색 메커니즘이 항상 더 나은 답으로 이어지지는 않는다는 뜻입니다.

## 검색-생성 갭: 왜 더 찾아도 더 좋아지지 않는가

논문이 식별한 가장 중요한 개념은 **"검색-생성 격차(retrieval-generation gap)"**입니다.

GraphRAG 시나리오 5는 검색 재현율(RoR, Retrieval of Retrieval)이 77.5%로 꽤 높습니다 — 정답 엔티티가 검색된 컨텍스트 어딘가에 존재합니다. 하지만 최종 답변의 Hit@1은 0.6422에 그칩니다. 정답이 컨텍스트에 있는데도 LLM이 그것을 답으로 뽑아내지 못하는 격차가 존재합니다.

왜 그럴까요? 논문은 세 가지 원인을 분석합니다.

**위치별 주의력 감쇠.** LLM이 성공적으로 추출한 엔티티는 프롬프트 내 평균 토큰 위치 1,888(중앙값 376)에 있었습니다. 반면 놓친 엔티티는 평균 5,534(중앙값 5,003) 위치에 있었습니다. 컨텍스트가 길어질수록 뒤쪽 정보는 "중간에 길을 잃는(lost in the middle)" 현상으로 무시됩니다.

**의미론적 중요성 편향.** LLM은 "정식"이거나 잘 알려진 엔티티를 우선시합니다. 임상 시나리오에서 주 질병은 정확히 식별하지만, 관련 하위 유형이나 복합 상태는 간결한 "감별 진단" 스타일을 선호해 무시합니다.

**기수성(cardinality) 기대.** 쿼리의 표현 방식이 답의 개수를 결정짓습니다. "이 약은 무엇인가…"라고 물으면, 컨텍스트에 5개의 유효 후보가 있어도 1개만 반환하는 경향이 있습니다.

이 격차는 "더 많이 검색하라"는 통상적 조언이 RAG 개선의 유효 전략이 아님을 시사합니다. 검색 데이터가 늘어날수록 노이즈와 lost-in-the-middle 효과가 실제 생성 품질을 저하시킬 수 있습니다.

## 토큰 절감: 컨텍스트 최적화의 효과

논문의 두 번째 공헌은 GraphRAG와 Agentic RAG의 토큰 비용을 줄이는 **컨텍스트 최적화 기법**입니다. 세 가지 전략을 제안합니다.

**1. 관계 그룹화 표현.** 전통적 삼중항 `(엔티티1 - 관계A - 엔티티2), (엔티티1 - 관계B - 엔티티2)`를 `(엔티티1 - (관계A, 관계B) - 엔티티2)`로 압축합니다. 같은 엔티티 쌍에 여러 관계가 있을 때 토큰을 크게 줄입니다.

**2. 그래프 검색 중복 제거.** 에이전트 세션 전체에 단일 통합 서브그래프를 유지하고, 새 검색 결과를 엔티티·관계 수준에서 중복 제거해 병합합니다. 컨텍스트가 선형이 아닌 아선형(sublinear)으로 자라게 합니다.

**3. 배치 에이전트 검색.** ReAct의 기본 동작(도구 호출마다 전체 대화 기록을 재전송) 대신, 에이전트가 여러 하위 쿼리를 한 번에 생성해 단일 배치로 실행합니다. ReAct와 ReWOO 스타일을 결합해 LLM 왕복 횟수를 줄입니다.

결과는 인상적입니다. 시나리오 5(GraphRAG 하이브리드)의 경우, 최적화 전 49.1K 토큰/쿼리에서 최적화 후 23.0K로 **53% 감소**했으며, 동시에 Hit@1은 0.6422→0.6330으로 거의 유지됐습니다. 토큰을 절약한 예산으로 검색을 확장(max 500 paths)하면 0.6422를 유지하면서 41% 절감 효과도 달성했습니다.

Agentic RAG(시나리오 8)에서는 기본 237K 토큰/쿼리에서 최적화 후 192K로 19% 감소했습니다. 시나리오 9(KG 도구 포함)에서는 최대 48% 절감을 기록했습니다.

즉, GraphRAG와 Agentic RAG는 기본적으로 토큰을 많이 소비하지만, 컨텍스트 엔지니어링만 제대로 하면 비용을 절반 가까이 줄이면서 품질을 유지할 수 있습니다.

## 실전 의사결정: 언제 무엇을 쓸 것인가

논문의 데이터와 업계 관행을 종합하면, 다음과 같은 의사결정 기준이 도출됩니다.

**시나리오 2(문서에 관계 정보 증강)로 시작하라.** 가장 높은 ROI를 제공합니다. 실시간 그래프 순회나 멀티턴 에이전트의 오버헤드 없이 관계형 컨텍스트의 이점을 얻습니다. 인덱싱 시점에만 관계를 텍스트로 직렬화해 넣으면 됩니다.

```python
# 의사코드: 문서에 1-hop 관계를 텍스트로 증강
def augment_with_relations(doc, kg):
    entity = doc.entity
    neighbors = kg.get_1hop_neighbors(entity)
    relation_text = format_relations_grouped(neighbors)
    # 엔티티1 - (관계A, 관계B) - 엔티티2 형태로 압축
    return f"{doc.text}\n\n[관계 정보]\n{relation_text}"

augmented_doc = augment_with_relations(doc, knowledge_graph)
vector_store.index(augmented_doc)  # 일반 벡터 검색으로 충분
```

**GraphRAG는 진짜 다중 홉 질문에만.** "A 시스템 장애가 어떤 조직과 서비스에 영향을 줬는가"처럼 여러 문서에 흩어진 관계를 연결해야 하는 질문, 또는 전체 코퍼스의 주제를 요약해야 하는 global question에만 GraphRAG가 의미가 있습니다. 이때도 시나리오 3(KG만)은 피하고, 시나리오 5(벡터+KG 하이브리드)를 쓰되 위 컨텍스트 최적화를 반드시 적용해야 합니다.

**Agentic RAG는 검색 자체가 추론을 필요로 할 때만.** 쿼리가 "A 고객의 Q3 재무와 B 고객의 건전성 지수를 비교해 이상치를 표시하라"처럼 중간 추론 없이는 무엇을 검색할지 결정할 수 없을 때 유효합니다. 단순 팩트 검색에 에이전트 루프를 돌리면 비용만 5-10배 늘고 답은 같습니다.

**비용 감각을 가져라.** 업계 추정치에 따르면 쿼리당 비용은 Naive RAG 약 $0.001, Advanced RAG(하이브리드+재랭크) $0.005-0.03, Agentic RAG $0.01-0.10, GraphRAG $0.02-0.15 수준입니다. GraphRAG의 인덱싱 비용은 단순 임베딩 대비 10-40배 높을 수 있어, 문서가 자주 바뀌는 환경에서는 유지보수 부담이 큽니다.

**Adaptive RAG를 목표로.** 가장 성숙한 2026년 프로덕션 패턴은 쿼리 복잡도를 분류해 적절한 파이프라인으로 라우팅하는 방식입니다. 단순 질문(대다수)은 빠르고 싼 Advanced RAG로, 복잡한 다단계 질문은 Agentic RAG로, 관계 합성 질문은 GraphRAG로 보냅니다. 모든 질문을 가장 비싼 경로로 몰지 마십시오.

## 주의할 점: 이 실험의 한계

논문의 결론을 맹신하면 안 됩니다. 몇 가지 한계가 있습니다.

첫째, **단일 도메인(정밀 의학)에서의 결과**입니다. STaRK-Prime은 엔티티(유전자, 단백질, 질병)와 관계가 비교적 정형화된 생물의학 도메인입니다. 법률 문서, 소스코드, 고객 지원 티켓처럼 관계가 더 비정형적인 도메인에서는 결과가 다를 수 있습니다.

둘째, **단일 LLM(Claude 3.7 Sonnet)**으로만 실험했습니다. 다른 모델, 특히 더 긴 컨텍스트 창을 가진 모델에서는 lost-in-the-middle 효과가 줄어들어 GraphRAG의 상대적 우위가 달라질 수 있습니다.

셋째, 2026년 4월 발표된 후속 연구 *"Do We Still Need GraphRAG?"* (arXiv:2604.09666)는 에이전트 검색이 GraphRAG와의 성능 격차를 "상당히 좁힌다"고 보고합니다. 다만 복잡한 다중 홉 추론에서는 GraphRAG가 여전히 안정적 우위를 보인다고 덧붙입니다. 즉, GraphRAG가 무조건 필요한 것은 아니지만, 특정 조건에서는 여전히 최선이라는 뉘앙스입니다.

## 핵심 요약

- **GraphRAG가 정답의 상위 호환이 아닙니다.** 문서에 1-hop 관계를 텍스트로 증강한 단순 RAG(시나리오 2)가 복잡한 GraphRAG·Agentic RAG를 Hit@1에서 능가하는 실험 결과가 있습니다.
- **검색-생성 격차를 경계하십시오.** 더 많이 검색한다고 답이 더 좋아지지 않습니다. 컨텍스트 길이와 LLM의 위치별 주의력 감쇠가 생성 품질의 병목입니다.
- **GraphRAG를 쓸 때는 컨텍스트 최적화가 필수입니다.** 관계 그룹화, 중복 제거, 배치 검색으로 토큰을 19-53% 줄이면서 품질을 유지할 수 있습니다.
- **복잡도는 실패를 관측했을 때만 올리십시오.** Advanced RAG(하이브리드 검색 + 재랭크)가 대부분의 프로덕션 기본값입니다. GraphRAG와 Agentic RAG는 측정된 실패 모드가 그것을 정당화할 때만 도입하십시오.

가장 먼저 시도할 것: 현재 RAG 파이프라인의 검색 실패를 RAGAS로 측정하고, 그중 관계형 질문 실패 비율이 유의미한지 확인하세요. 그 비율이 높다면 시나리오 2(문서 관계 증강)부터 시작하고, 그래도 부족하면 GraphRAG 하이브리드 + 컨텍스트 최적화로 올라가십시오. 비용 대비 효과가 가장 좋은 경로입니다.
