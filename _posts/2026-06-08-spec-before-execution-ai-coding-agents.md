# AI 코딩 에이전트는 왜 스펙부터 써야 하나

요즘 AI 코딩 도구를 쓰다 보면 가장 먼저 느끼는 장점은 속도다. 요구사항을 한 번 던지면 코드 초안을 빠르게 만들고, 테스트도 돌리고, 파일도 고친다. 그런데 바로 그 속도 때문에 팀은 자주 같은 문제를 겪는다. 모델이 “무엇을 만들지”보다 “어떻게 빨리 만들지”에 먼저 반응하면서, 숨은 가정이 쌓이고, 리뷰 비용이 뒤늦게 폭발한다.

그래서 최근에 다시 주목받는 방식이 **Spec-Driven Development(SDD)** 다. GitHub Spec Kit은 이를 “AI-assisted software development의 중심에 specification을 두는 방식”으로 설명하고, 핵심 흐름을 **Spec → Plan → Tasks → Implement** 로 정리한다. 더 중요한 점은 각 단계가 다음 단계를 위한 **Markdown 산출물**을 만든다는 것이다. 즉, 프롬프트 한 번으로 끝나는 작업이 아니라, 사람이 검토할 수 있는 작업 기록이 남는다.

## 1. 프롬프트는 요구사항이 아니라 실행 힌트다

LLM에게 “로그인 기능 만들어줘”라고 말하면 모델은 대체로 그럴듯한 코드를 낸다. 하지만 그 한 문장에는 이미 수많은 질문이 생략돼 있다.

- 비밀번호 정책은 무엇인가
- 세션 기반인지, 토큰 기반인지
- 실패 시 잠금 정책이 있는가
- 어떤 예외는 허용하고 어떤 예외는 막아야 하는가
- 성공 기준을 어떻게 확인할 것인가

이 질문이 풀리지 않은 상태에서 바로 코드를 생성하면, 에이전트는 빈칸을 추측으로 채운다. 스펙은 이 빈칸을 먼저 채우는 장치다. “무엇을 만들지”를 먼저 적어 두면, 모델은 코드를 쓰기 전에 제약과 목표를 읽는다. GitHub Spec Kit의 표현을 빌리면, AI 코딩 에이전트에 **structured context** 를 주는 셈이다.

## 2. 실전 워크플로우: spec.md → plan.md → tasks.md

나는 이런 식으로 폴더를 나눈다.

```text
feature-x/
  spec.md
  plan.md
  tasks.md
  notes.md
```

### spec.md에 넣을 것
- 문제 정의
- 사용자 시나리오
- 범위 밖 항목
- 성공 기준
- 성능/보안/호환성 제약

### plan.md에 넣을 것
- 구현 전략
- 변경될 파일 목록
- 위험 요소
- 테스트 전략

### tasks.md에 넣을 것
- 작은 단위의 체크박스
- 의존성 순서
- 수동 검증 항목

예시:

```md
## spec
- 사용자는 이메일과 비밀번호로 로그인한다.
- 실패 5회 이후에는 계정이 일시 잠긴다.
- 성공 시 /dashboard 로 이동한다.

## acceptance criteria
- [ ] 유효하지 않은 자격 증명은 401을 반환한다.
- [ ] 잠금 상태에서는 423을 반환한다.
- [ ] 로그인 성공 후 세션이 생성된다.
```

이렇게 쓰면 에이전트는 “무엇을 고칠지”보다 “무엇을 만족해야 하는지”를 먼저 보게 된다. Claude Code처럼 코드베이스를 읽고 파일을 수정하며 명령을 실행하는 도구일수록, 이 단계 분리가 효과가 크다. 도구가 강력할수록 입력 품질이 더 중요해지기 때문이다.

## 3. 보안 관점에서 스펙이 중요한 이유

스펙은 생산성 도구이기도 하지만, 동시에 보안 경계이기도 하다. 최근 연구는 **agentic coding assistants** 에서의 prompt injection이 단순한 텍스트 오염이 아니라, skills, tools, protocol ecosystems 전반의 취약점으로 이어질 수 있음을 다룬다. 다시 말해, 에이전트가 외부 입력을 읽고 동시에 도구를 호출할 수 있으면, 잘못된 지시가 “출력” 수준에서 끝나지 않고 “행동”으로 번질 수 있다.

그래서 실무에서는 다음 원칙이 유효하다.

1. **읽기 단계와 실행 단계를 분리한다**
2. **처음에는 읽기 전용으로 시작한다**
3. **쉘, 네트워크, 파일쓰기 권한은 최소화한다**
4. **스펙 승인 전에는 실제 코드 변경을 막는다**
5. **diff와 테스트 결과를 사람에게 보여 준 뒤 다음 단계로 넘긴다**

이건 과도한 보수주의가 아니다. 에이전트가 똑똑해질수록, 잘못된 입력이 더 빨리, 더 넓게 퍼질 수 있기 때문이다.

## 4. 팀에서 바로 쓸 수 있는 가드레일

아래처럼 간단한 규칙만 넣어도 품질이 크게 안정된다.

```yaml
agent_policy:
  phase_1: spec_only
  phase_2: plan_review
  phase_3: task_breakdown
  phase_4: implementation
  allowed_actions:
    - read
    - summarize
    - propose_patch
  blocked_without_approval:
    - write
    - delete
    - network
    - shell_exec
```

그리고 PR 템플릿에도 한 줄을 추가한다.

```md
- [ ] spec.md 승인 완료
- [ ] acceptance criteria 정의 완료
- [ ] 위험 요소와 롤백 계획 확인
- [ ] 테스트 명령어 기록 완료
```

이 체크리스트 하나로도 “코드부터 쓰고 나중에 맞춘다”는 습관을 줄일 수 있다.

## 결론: 에이전트를 느리게 만드는 게 아니라, 신뢰 가능하게 만드는 일

핵심은 간단하다.

- 스펙은 프롬프트보다 오래 남는 작업 산출물이다.
- Plan과 Tasks는 모델의 추측을 줄여 준다.
- 에이전트가 강력할수록 실행 권한은 늦게 줘야 한다.
- 보안 문제는 종종 모델이 아니라 **경계 설계** 에서 생긴다.
- GitHub Spec Kit처럼 스펙을 먼저 쓰는 흐름은 AI 코딩의 속도를 늦추기보다, 오히려 재작업을 줄여 전체 속도를 높인다.

오늘 당장 해볼 일은 하나면 충분하다. 다음 기능 요청을 받으면 코드 생성부터 시키지 말고, 먼저 `spec.md`를 한 페이지로 써 보자. 그다음에야 계획과 태스크로 나누고, 마지막에 구현을 맡기면 된다. AI 에이전트 시대의 생산성은 더 빠른 답변이 아니라, 더 나은 경계에서 나온다.
