# LLM 비용을 90% 줄이는 프롬프트 캐싱: 세 제공자 실전 비교와 4가지 함정

매일 수만 건의 LLM API 요청을 보내는 서비스에서, 시스템 프롬프트와 도구 정의가 20,000토큰을 넘는 건 흔하다. 에이전트 워크플로우는 동일한 프롬프트 접두사(prefix)를 수십 번 반복 전송하고, RAG 파이프라인은 검색된 문서를 매 요청마다 다시 보낸다. 이 토큰들은 이미 한 번 처리된 것들이다 — 하지만 캐싱하지 않으면 매번 처음부터 다시 계산하고, 전액을 청구받는다.

프롬프트 캐싱(Prompt Caching)은 이 낭비를 없앤다. 모델이 이미 계산한 키-값(KV) 텐서를 저장해두고, 다음 요청이 같은 접두사로 시작하면 그 부분의 연산을 건너뛴다. 출력은 바이트 단위로 동일하고, 비용만 줄어든다. 양자화도 아니고, 증류도 아닌 — 단지 프롬프트 구조를 바꾸는 것만으로 입력 비용의 최대 90%를 절약할 수 있다.

## 프롬프트 캐싱과 시맨틱 캐싱은 다르다

혼동하기 쉬운 두 개념부터 구분해야 한다.

| | 프롬프트 캐싱 | 시맨틱 캐싱 |
|---|---|---|
| **매칭 방식** | 정확한 접두사 일치 (exact prefix match) | 의미적 유사도 (semantic similarity) |
| **저장 단위** | KV 텐서 (GPU 메모리) | 임베딩 벡터 + 응답 |
| **출력 보장** | 동일 (deterministic) | 유사할 뿐, 부정확 가능 |
| **위치** | 제공자 인프라 내부 | 애플리케이션 레이어 |
| **대표 도구** | Anthropic cache_control, OpenAI 자동 캐싱 | GPTCache, Redis + 임베딩 |

프롬프트 캐싱은 "같은 접두사면 같은 연산 결과를 재사용"한다는 정확한 매칭이므로, 모델 출력이 달라질 걱정이 없다. 반면 시맨틱 캐싱은 "비슷한 질문엔 비슷한 답을 반환"하므로, 임계값 설정에 따라 오답을 반환할 수 있다. 비용 절감 효과는 프롬프트 캐싱이 더 크고 안전하지만, 접두사가 정확히 일치해야 한다는 제약이 있다.

## 작동 원리: KV 캐시와 접두사 매칭

트랜스포머 모델이 프롬프트를 처리할 때, prefill 단계에서 각 토큰의 키(key)와 값(value) 텐서를 계산한다. 이 KV 캐시는 어텐션 연산을 효율적으로 만드는 핵심 데이터 구조다. 프롬프트 캐싱은 이 KV 캐시의 앞부분을 저장해두고, 다음 요청이 같은 접두사로 시작하면 그 부분의 prefill 연산을 건너뛴다.

중요한 것은 매칭이 **순차적이고 정확하다**는 점이다. 프롬프트가 `시스템 프롬프트 → 컨텍스트 → 사용자 질의` 순서라면, 시스템 프롬프트의 어느 한 글자라도 바뀌면 그 이후의 모든 캐시가 무효화된다. 따라서 변하지 않는 내용을 앞에, 매번 바뀌는 내용을 뒤에 배치해야 한다.

## 세 제공자 비교: 가격 구조와 TTL

세 제공자 모두 프롬프트 캐싱을 지원하지만, 구현 방식과 경제 구조가 다르다. 아래 수치는 각사 공식 문서를 기반으로 한다.

### Anthropic (Claude)

가장 세밀한 제어를 제공한다. 명시적 캐시 중단점(explicit breakpoint)을 프롬프트 어디에나 배치할 수 있고, TTL을 5분 또는 1시간 중 선택할 수 있다.

**가격 배수 (Anthropic 공식 문서 기준):**

| 항목 | 배수 (기본 입력 대비) |
|---|---|
| 5분 캐시 쓰기 | 1.25× (25% 프리미엄) |
| 1시간 캐시 쓰기 | 2.0× (100% 프리미엄) |
| 캐시 읽기 | 0.1× (90% 할인) |

최소 1,024토큰 이상의 접두사만 캐시 대상이 된다. 캐시는 사용될 때마다 TTL이 갱신되므로, 5분 간격으로 요청이 들어오면 추가 비용 없이 캐시가 유지된다.

### OpenAI

GPT-5.6 모델 패밀리부터 구조가 바뀌었다. 이전에는 캐시 쓰기에 추가 비용이 없었지만, GPT-5.6 이상부터는 캐시 쓰기에 1.25×의 비용이 부과된다. 대신 캐시 매칭이 더 일관적이고 신뢰할 수 있게 개선되었다.

- **캐시 쓰기**: GPT-5.6+ 모델은 1.25× (이전 모델은 추가 비용 없음)
- **캐시 읽기**: 할인된 가격 (cached-input rate)
- **최소 토큰**: 1,024토큰
- **TTL**: GPT-5.6+ 모델은 기본 30분 (현재 지원되는 유일한 값)
- **자동 캐싱**: 코드 변경 없이 자동 적용. `prompt_cache_key` 파라미터로 캐시 라우팅을 제어할 수 있다
- **명시적 중단점**: GPT-5.6+에서 `prompt_cache_breakpoint`로 정밀 제어 가능. 요청당 최대 4개 중단점

### Google Gemini

컨텍스트 캐싱(Context Caching)이라는 이름으로 제공한다. Gemini 2.5 이상 모델에서는 암시적 캐싱(implicit caching)이 기본 활성화되어 있어, 별도 설정 없이 같은 접두사가 감지되면 자동으로 비용이 절감된다. 명명된 캐시 객체(Named Cache Object)를 명시적으로 생성하는 방식도 지원한다. 캐시된 토큰은 일반 입력보다 낮은 가격으로 청구되며, 최소 입력 토큰 수와 저장 비용은 모델별로 다르므로 공식 문서를 확인해야 한다.

## 손익분기점 계산: 캐시가 돈을 버는 순간

프롬프트 캐싱이 항상 이익인 것은 아니다. 캐시 쓰기에 프리미엄이 붙기 때문에, 캐시 적중률(hit rate)이 너무 낮으면 오히려 비용이 늘어난다.

Anthropic의 5분 캐시를 예로 들어 손익분기점을 계산해보자.

- 캐시 쓰기: 1.25× (한 번)
- 캐시 읽기: 0.1× (반복)

캐시를 N번 읽으면, 총 비용은 `1.25 + 0.1×N`이다. 캐싱 없이 N+1번 전체 입력을 처리하면 `1.0×(N+1)`이다. 두 값이 같아지는 지점이 손익분기점이다:

```
1.25 + 0.1N = 1.0 + 1.0N
0.25 = 0.9N
N ≈ 0.28
```

즉, **캐시를 단 한 번이라도 재사용하면 본전 이상**이 된다. 하지만 이는 캐시가 유효한 5분 안에 재사용됐을 때의 이야기다. TTL이 만료되면 캐시는 사라지고, 다음 요청은 다시 1.25×로 새 캐시를 써야 한다. 따라서 실제 손익분기점은 **5분 안에 최소 1~2회 이상의 재사용**이 일어나는지에 달려 있다.

```python
# Anthropic 손익분기점 시뮬레이션
def cached_cost(writes, reads, base_per_million=3.0):
    """Claude Opus 기준 (예시)"""
    write_cost = writes * base_per_million * 1.25
    read_cost = reads * base_per_million * 0.1
    return write_cost + read_cost

def uncached_cost(total_calls, base_per_million=3.0):
    return total_calls * base_per_million

# 시스템 프롬프트 10K 토큰, 하루 1000회 호출
writes = 1   # 캐시 갱신 주기마다 1회 쓰기
reads = 100  # 그 후 100회 읽기
print(f"캐싱 적용: ${cached_cost(writes, reads) / 100:.4f}/1K calls")
print(f"캐싱 미적용: ${uncached_cost(101) / 100:.4f}/1K calls")
# 캐싱 적용: 약 89% 절감
```

## 실전 구현: Anthropic 명시적 캐싱

Anthropic SDK로 명시적 캐시 중단점을 설정하는 예시다.

```python
import anthropic

client = anthropic.Anthropic()

# 안정적인 접두사에 캐시 중단점 설정
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "당신은 보안 코드 분석 전문가입니다. 다음 규칙을 따르세요: ...",
            # 이 블록까지의 접두사를 캐시 (기본 TTL 5분)
            "cache_control": {"type": "ephemeral"}
        }
    ],
    messages=[
        {"role": "user", "content": "이 함수의 취약점을 분석해줘: ..."}
    ]
)

# 캐시 적중 여부 확인
usage = response.usage
print(f"캐시 읽기 토큰: {usage.cache_read_input_tokens}")
print(f"캐시 쓰기 토큰: {usage.cache_creation_input_tokens}")
print(f"일반 입력 토큰: {usage.input_tokens}")
```

핵심은 `cache_control`을 **변하지 않는 블록의 끝**에 배치하는 것이다. 시스템 프롬프트에 타임스탬프나 요청 ID가 포함되어 있으면, 매 요청마다 접두사가 달라져 캐시가 무효화된다.

## OpenAI 자동 캐싱: 설정 없이 적용되지만

OpenAI는 코드 변경 없이 자동으로 캐싱을 적용한다. 요청이 들어오면 프롬프트 접두사의 해시를 기반으로 같은 서버로 라우팅하고, 그 서버에 캐시가 있으면 할인된 가격으로 처리한다.

```python
from openai import OpenAI

client = OpenAI()

# 자동 캐싱: 코드 변경 불필요
# prompt_cache_key로 라우팅 제어 가능 (GPT-5.6+ 권장)
response = client.chat.completions.create(
    model="gpt-5.6",
    messages=[
        {"role": "system", "content": LARGE_SYSTEM_PROMPT},  # 1,024+ 토큰
        {"role": "user", "content": user_query}
    ],
    extra_body={
        "prompt_cache_key": "tenant:acme:kb-v1"  # 캐시 라우팅 키
    }
)

# 캐시 적중 확인 (usage 객체)
cached = response.usage.prompt_tokens_details.cached_tokens
print(f"캐시된 토큰: {cached}")
```

`prompt_cache_key`는 같은 접두사를 공유하는 요청들을 같은 캐시로 라우팅하는 역할을 한다. 멀티테넌트 환경에서 테넌트별로 키를 나누면 적중률을 높일 수 있다. 단, 한 키당 약 15회/분을 넘기면 캐시 적중이 누락될 수 있으므로, 고트래픽에서는 키를 분산해야 한다.

## 4가지 함정: 캐시 적중률을 죽이는 실수들

### 1. 프롬프트 접두사에 동적 데이터 넣기

가장 흔한 실수다. 시스템 프롬프트 첫 줄에 `"현재 시간: 2026-07-13T14:23:11"` 같은 타임스탬프를 넣으면, 매 초마다 접두사가 달라져 캐시가 완전히 무효화된다. 요청 ID, 사용자별 세션 정보, 캐시된 검색 결과도 마찬가지다.

**해결**: 동적 데이터는 항상 프롬프트의 **마지막**, 즉 사용자 메시지 안에 넣어라. 정적 콘텐츠(도구 정의 → 시스템 프롬프트 → 참조 문서)를 앞에 배치하고, 동적 콘텐츠(대화 기록 → 현재 질의)를 뒤에 배치한다.

### 2. 최소 토큰 미달

세 제공자 모두 캐시 대상 접두사가 일정 토큰 수 이상이어야 한다. Anthropic과 OpenAI는 1,024토큰이 최소 기준이며, Gemini는 모델별로 최소 토큰 수가 다르다. 시스템 프롬프트가 800토큰밖에 안 된다면, 캐싱 대상 자체가 되지 않을 수 있다. 이 경우 Few-shot 예제를 추가하거나 도구 정의를 더 포함해 접두사를 늘리는 것이 캐시 혜택을 받는 첫걸음이다.

### 3. TTL 만료 간과

5분 캐시는 마지막 사용 후 5분이 지나면 사라진다. 사용자가 10분 간격으로 질문하는 챗봇이라면, 5분 캐시는 매번 새로 써야 한다. 이런 경우 Anthropic의 1시간 TTL(2× 쓰기 비용)이 더 나을 수 있다. OpenAI의 GPT-5.6+는 기본 30분 TTL로, 중간 강도 워크로드에 적합하다.

### 4. 적중률 모니터링 누락

캐시 적중률이 30% 미만이라면 구조적 문제가 있다는 뜻이다. 안정적인 접두사를 가진 워크로드라면 60% 이상이 정상이다. 매 요청의 `cache_read_input_tokens` / `input_tokens` 비율을 로깅하고, 이것이 갑자기 떨어지면 프롬프트 구조가 바뀌었는지 확인해야 한다.

## 핵심 요약

- **프롬프트 캐싱은 시맨틱 캐싱과 다르다** — 정확한 접두사 매칭으로 출력이 동일하게 보장된다
- **비용 효과는 압도적이다** — Anthropic은 캐시 읽기 시 90% 할인, OpenAI는 자동 적용
- **배치가 전부다** — 정적 → 동적 순서로 프롬프트를 구성하면 적중률이 극적으로 바뀐다. 한 사례에서는 동적 작업 메모리를 시스템 프롬프트에서 분리해 적중률을 7%에서 84%로 끌어올렸다 (업계 사례 보고 기준)
- **손익분기점은 낮다** — 5분 안에 1회 재사용만 돼도 본전. 단, TTL 만료에 주의
- **첫 단계**: 지금 사용 중인 API의 usage 객체에서 `cache_read` 필드를 확인하라. 0이라면, 아직 캐싱을 받지 못하고 있는 것이다

> **참고**: 본문의 가격 배수(1.25×, 2.0×, 0.1×)는 Anthropic 공식 문서(2026-07 기준)에서 직접 확인했다. OpenAI의 캐시 읽기 할인율은 모델에 따라 다르며, 공식 문서에서는 "cached-input rate"로 표기된다. 구체적인 달러 단위 가격은 각사 프라이싱 페이지를 참조하라.
