# 에이전트가 멈추지 않게 하려면: 백그라운드 실행, 샌드박스, MCP 레지스트리를 분리하라

대부분의 에이전트 실패는 모델 성능이 부족해서가 아니라, **한 번에 너무 많은 책임을 채팅창 하나에 몰아넣어서** 생긴다. 긴 추론은 오래 걸리고, 파일 작업은 권한이 필요하고, 외부 도구 연결은 탐색과 보안이 필요하다. 이 세 가지를 섞어버리면 사용자는 “왜 멈췄지?”를 묻게 된다.

그래서 요즘은 에이전트를 하나의 거대한 챗봇으로 보기보다, 역할별로 분리해서 설계하는 편이 훨씬 낫다. 핵심은 간단하다. **추론은 백그라운드로**, **실행은 샌드박스로**, **도구 발견은 레지스트리로** 나누는 것이다.

## 1) 백그라운드는 “기다림”을 상태로 바꾼다

OpenAI의 Background mode는 긴 작업을 비동기로 돌리기 위한 기능이다. 공식 문서에서는 오래 걸리는 작업을 백그라운드에서 실행하고, 응답 객체를 폴링해서 상태를 확인할 수 있다고 설명한다. 예시로는 `background: true`를 넣어 Responses API를 호출하는 패턴이 제시된다.

이 방식의 장점은 분명하다.
- 사용자는 타임아웃을 신경 쓰지 않아도 된다
- 모델이 몇 분이 걸리든 작업 흐름이 끊기지 않는다
- 결과가 아직 안 나왔을 때도 “진행 중”이라는 상태를 시스템이 유지한다

다만 주의점도 있다. 공식 문서에 따르면 백그라운드 모드는 응답 데이터를 **대략 10분 정도** 보관해 폴링을 가능하게 하며, 이 때문에 **Zero Data Retention(ZDR)과는 호환되지 않는다**. 즉, “그냥 오래 돌린다”가 아니라 “오래 돌리는 대신 어떤 데이터 경계가 생기는가”까지 같이 봐야 한다.

### 예시: OpenAI Responses API를 백그라운드로 실행하기

```python
from openai import OpenAI

client = OpenAI()

resp = client.responses.create(
    model="gpt-5.5",
    input="긴 코드베이스를 읽고 리팩터링 계획을 작성해줘.",
    background=True,
)

print(resp.status)
```

실전에서는 여기서 끝나지 않는다. `resp.status`를 저장해 두고, 주기적으로 폴링하면서 완료 여부를 확인해야 한다. 중요한 건 “한 번에 답을 받는 API”로 생각하지 말고, **작업 큐에 넣는 인터페이스**로 이해하는 것이다.

## 2) 샌드박스는 “할 수 있는 일”을 줄여서 안전을 만든다

Claude Code 문서의 보안 섹션은 더 직접적이다. 기본 보호 장치로 **Sandboxed bash tool**을 제공하고, `sandbox` 명령으로 파일시스템과 네트워크 격리를 설정할 수 있다고 안내한다. 또 Claude Code는 기본적으로 시작한 폴더와 그 하위 폴더에만 쓰기 가능하며, 상위 디렉터리는 명시적 허가가 없으면 수정할 수 없다.

이건 에이전트에게 불편함을 주는 기능이 아니라, 오히려 **사고를 줄이는 구조적 장치**다. 특히 다음 상황에서는 샌드박스가 필수에 가깝다.
- 외부 패키지 설치가 필요한 작업
- 실수로 `rm -rf` 같은 파괴적 명령이 나갈 수 있는 작업
- 네트워크 접근 여부를 통제해야 하는 작업

### 예시: Claude Code에서 샌드박스 경계 열기

```bash
/sandbox
```

샌드박스의 의미는 단순하다. 에이전트가 “무슨 일이든 다 할 수 있는 상태”가 아니라, **정해진 디렉터리와 네트워크 규칙 안에서만 움직이는 상태**로 만드는 것이다. 이 제약이 있어야 로그를 믿을 수 있고, 실패 원인도 추적할 수 있다.

## 3) MCP 레지스트리는 “도구 목록”을 사람 대신 관리한다

에이전트가 커질수록 문제는 실행보다 **발견**에서 먼저 터진다. 어떤 MCP 서버가 있고, 최신 버전은 무엇이며, 무엇을 연결해도 되는지 사람이 기억하기 시작하면 운영이 금세 꼬인다.

이때 유용한 것이 Official MCP Registry다. 공식 레지스트리는 MCP 서버를 발견할 수 있게 해 주는 허브 역할을 한다. 쉽게 말해, 에이전트가 쓸 수 있는 도구를 “문서 한 장”이 아니라 **조회 가능한 카탈로그**로 바꾸는 것이다.

이 구조가 좋은 이유는 세 가지다.
- 연결 가능한 서버를 한곳에서 찾을 수 있다
- 버전과 상태를 별도로 관리할 수 있다
- 팀 내부에서 “이 도구를 써도 되는가?”라는 질문을 줄일 수 있다

## 4) 실전 아키텍처: 한 작업을 세 층으로 쪼개기

내가 추천하는 기본 분리는 다음과 같다.

1. **컨트롤러**: 사용자의 요청을 받아 백그라운드 작업으로 넘긴다
2. **실행기**: 샌드박스 안에서 코드 실행, 파일 수정, 테스트를 수행한다
3. **도구 레이어**: MCP 레지스트리에서 필요한 서버를 찾아 연결한다

이렇게 나누면 장점이 뚜렷하다. 백그라운드는 시간 문제를 해결하고, 샌드박스는 권한 문제를 해결하고, 레지스트리는 발견 문제를 해결한다. 세 문제를 한 덩어리로 풀려고 하면 결국 “잘 되는 것처럼 보이다가 어느 날 깨지는” 시스템이 된다.

### 추천 작업 흐름

- Step 1: 사용자의 요청을 백그라운드 작업으로 등록한다
- Step 2: 실행 전 샌드박스 범위를 고정한다
- Step 3: 필요한 MCP 서버만 레지스트리에서 가져온다
- Step 4: 완료 후 상태와 산출물을 분리해서 저장한다
- Step 5: 실패하면 재시도보다 먼저 권한·경계·도구 목록을 확인한다

## 결론: 에이전트는 똑똑함보다 경계가 중요하다

정리하면, 좋은 에이전트는 “더 많은 걸 할 수 있는 모델”이 아니라 **언제 기다리고, 어디서 실행하고, 어떤 도구를 쓸지 분리된 시스템**이다.

- 긴 작업은 채팅이 아니라 백그라운드로 돌린다
- 실제 실행은 샌드박스 안에 가둔다
- 도구 연결은 MCP 레지스트리처럼 조회 가능한 구조로 둔다
- 타임아웃, 권한, 발견 문제를 한 번에 해결하려 하지 않는다
- 에이전트의 품질은 응답 문장보다 운영 경계에서 갈린다

오늘 바로 해볼 첫 단계는 간단하다. 지금 돌고 있는 에이전트 작업 하나를 골라서, **추론·실행·도구 발견** 중 무엇이 병목인지 나눠 적어보자. 그 다음에야 비로소 자동화가 안정화된다.
