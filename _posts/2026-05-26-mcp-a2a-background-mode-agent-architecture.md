# MCP는 도구, A2A는 협업: 에이전트 아키텍처를 나누는 법

요즘 에이전트 시스템을 만들다 보면 같은 질문을 반복해서 마주칩니다. "이 작업은 한 에이전트가 끝까지 처리해야 할까, 아니면 다른 에이전트에게 넘겨야 할까?" "도구 연동은 어디까지 프로토콜로 묶고, 어디서부터는 별도 작업으로 분리해야 할까?"

이 질문에 대한 가장 실용적인 답은, **MCP, A2A, Background mode를 같은 층위로 보지 않는 것**입니다. MCP는 주로 에이전트가 도구와 리소스에 연결되는 경로이고, A2A는 서로 다른 에이전트가 협업하는 경로입니다. 여기에 Background mode를 더하면, 오래 걸리는 작업을 요청 경로에서 떼어내고 상태 기반으로 관리할 수 있습니다.

즉, 설계의 핵심은 "무엇을 모델에게 맡길까"가 아니라 **"무엇을 도구로 처리하고, 무엇을 다른 에이전트에게 넘기고, 무엇을 비동기로 돌릴까"** 입니다.

## 1) 세 프로토콜의 역할을 분리하면 구조가 선명해진다

A2A 공식 문서는 A2A를 **서로 다른 AI 에이전트가 안전하게 통신하고 협업할 수 있게 하는 open standard**로 설명합니다. 또한 A2A의 핵심 개념에는 Agent Card, Task, Message, Part가 포함됩니다. 여기서 특히 중요한 점은, 원격 에이전트가 클라이언트 입장에서는 **opaque한 블랙박스**처럼 보인다는 것입니다. 내부 메모리나 툴을 노출하지 않아도 됩니다.

반면 MCP는 A2A 문서에서도 **agent-to-tool communication**을 담당하는 보완 표준으로 소개됩니다. 다시 말해, MCP는 "이 에이전트가 어떤 API와 리소스를 쓸 수 있는가"를 정리하는 층이고, A2A는 "이 에이전트가 다른 에이전트와 어떻게 일감을 주고받는가"를 다루는 층입니다.

여기에 OpenAI의 Background mode를 더하면, 긴 작업을 요청-응답 한 번으로 끝내지 않고 **비동기 작업 + 상태 폴링** 패턴으로 바꿀 수 있습니다. OpenAI 문서는 Background mode를 오래 걸리는 작업을 백그라운드에서 비동기로 실행하고, 응답 객체를 폴링해 상태를 확인하는 방식으로 설명합니다.

정리하면 이렇게 볼 수 있습니다.

- **MCP**: 도구, API, 리소스 연결
- **A2A**: 에이전트 간 협업과 작업 위임
- **Background mode**: 긴 작업의 비동기 처리와 상태 추적

## 2) 실전에서는 이렇게 라우팅하는 게 좋다

아래처럼 규칙을 나누면 구조가 단순해집니다.

```yaml
routing:
  mcp:
    when: "외부 API, 사내 DB, 파일 시스템, 검색 인덱스가 필요할 때"
    goal: "도구 호출을 표준화한다"
  a2a:
    when: "작업이 독립된 책임 단위이거나 다른 에이전트의 전문성이 더 적합할 때"
    goal: "에이전트에게 업무를 위임한다"
  background_mode:
    when: "응답 시간 안에 끝나지 않거나 상태 추적이 필요한 장기 작업일 때"
    goal: "요청 경로를 막지 않고 나중에 회수한다"
```

이 규칙을 실제 서비스에 적용하면, 한 에이전트가 모든 일을 떠안는 구조를 피할 수 있습니다. 예를 들어:

1. **MCP로 사실 수집**: 문서 검색, 사내 위키 조회, 로그 조회
2. **A2A로 작업 분리**: 번역, 코드 리뷰, 리포트 작성처럼 독립된 전문 작업을 다른 에이전트에 위임
3. **Background mode로 장기 실행**: 대규모 요약, 긴 문서 생성, 대량 분석처럼 오래 걸리는 작업을 비동기로 처리

이 방식의 장점은 디버깅도 쉬워진다는 점입니다. 문제가 생겼을 때 "모델이 왜 틀렸나"만 보지 않고, "도구 계층의 문제인지", "위임 경계의 문제인지", "비동기 상태 관리의 문제인지"를 분리해서 볼 수 있습니다.

## 3) 바로 써먹는 구현 예시

OpenAI Background mode의 공식 예시는 `client.responses.create(..., background=True)`로 응답을 생성한 뒤, `resp.status`가 `queued` 또는 `in_progress`인 동안 `client.responses.retrieve(resp.id)`를 반복 호출하는 구조입니다. 이 패턴은 장기 작업에 특히 잘 맞습니다.

```python
from time import sleep
from openai import OpenAI

client = OpenAI()

resp = client.responses.create(
    model="YOUR_MODEL",
    input="문서 100개를 요약해서 운영 리포트를 만들어줘.",
    background=True,
)

while resp.status in {"queued", "in_progress"}:
    print(f"Current status: {resp.status}")
    sleep(2)
    resp = client.responses.retrieve(resp.id)

print(f"Final status: {resp.status}\nOutput:\n{resp.output_text}")
```

이 코드를 에이전트 시스템에 넣을 때는 다음 두 가지를 같이 붙이는 게 좋습니다.

```python
job = {
    "task_type": "report_generation",
    "handoff": "a2a",
    "tools": ["mcp://search", "mcp://docs", "mcp://db"],
    "async": True,
}
```

이런 식으로 메타데이터를 남기면, 나중에 관측 도구에서 "어떤 작업이 왜 A2A로 넘어갔는지"를 추적하기 쉽습니다. 특히 A2A의 Task는 상태가 있는 작업 단위이므로, 단순 메시지보다 운영 측면에서 더 다루기 편합니다.

## 4) 보안은 MCP에서 특히 먼저 잡아야 한다

MCP 공식 보안 문서는 prompt injection, session hijack, local server 공격 같은 위협을 다룹니다. 그래서 MCP 서버를 붙일 때는 "일단 연결되니까 괜찮다"가 아니라, **권한 범위를 최소화하고, 샌드박싱과 사용자 동의를 분명히 하고, 토큰/세션 경계를 검증하는 것**부터 시작해야 합니다.

실무적으로는 다음 원칙이 유효합니다.

- 도구 서버는 필요한 권한만 주기
- 로컬 서버도 신뢰 경계를 갖기
- 사용자 동의가 필요한 작업은 명시적으로 분리하기
- 에이전트가 읽은 외부 콘텐츠를 곧바로 도구 실행 지시로 해석하지 않기

이건 단순한 보안 체크리스트가 아니라, **프로토콜을 어디에 쓰고 어디에 쓰지 않을지 결정하는 설계 원칙**입니다.

## 결론: 한 에이전트가 다 하지 말고, 경계를 나눠라

에이전트 시스템을 안정적으로 만들고 싶다면 다음 순서를 추천합니다.

- **도구 연동은 MCP로 표준화**한다
- **에이전트 간 협업은 A2A로 위임**한다
- **오래 걸리는 작업은 Background mode로 비동기화**한다
- **보안은 MCP 경계부터 최소 권한으로 설계**한다
- **모든 작업을 하나의 프롬프트 체인에 몰아넣지 않는다**

첫 단계로는, 지금 운영 중인 에이전트 하나를 골라서 "도구 호출", "타 에이전트 위임", "장기 작업"을 각각 어디서 처리하는지 표로 나눠보세요. 그 표가 생기면, 아키텍처는 훨씬 덜 흔들립니다.
