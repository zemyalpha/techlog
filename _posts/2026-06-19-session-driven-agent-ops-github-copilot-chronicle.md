# 세션이 제품이 됐다: GitHub Copilot 앱과 /chronicle가 바꾸는 코딩 에이전트 운영

코딩 에이전트의 경쟁력은 더 좋은 한 줄 프롬프트를 찾는 데서 끝나지 않는다. 지금 중요한 건 **작업을 어떤 단위로 저장하고, 다시 불러오고, 이어서 실행하느냐**다. 최근 GitHub Copilot 앱, `/chronicle`, Copilot Chat의 변화는 그 단위가 이제 채팅이 아니라 **세션(session)** 이라는 점을 분명히 보여준다.

이 흐름이 의미 있는 이유는 단순하다. 프롬프트는 순간적이지만 세션은 누적된다. 누적된 세션은 브랜치, 파일 변경, 도구 사용, 후속 질문까지 포함한 작업의 맥락을 남긴다. 에이전트를 운영한다는 건 결국 이 맥락을 얼마나 잘 다루느냐의 문제다.

## 1) GitHub가 말하는 새 기본값: 세션

GitHub Copilot 앱은 이제 macOS, Windows, Linux에서 **일반 공개(GA)** 상태다. GitHub는 이 앱을 "agent-driven development"를 위한 데스크톱 홈으로 설명한다. 또 문서에서는 여러 개의 **격리된 agent session**을 동시에 돌릴 수 있고, 각 세션이 자체 브랜치를 가진다고 안내한다.

이건 꽤 큰 변화다. 예전에는 한 채팅창에서 이것저것 섞어 물어보는 방식이 흔했지만, 이제는 세션 자체가 작업의 경계가 된다. 즉, 버그 수정 세션, 리팩터링 세션, 문서 정리 세션을 분리하고 각각을 독립된 결과물로 관리하는 편이 훨씬 안전하다.

GitHub가 `/chronicle`를 통해 세션 히스토리를 검색 가능하게 만들고, Copilot Chat이 과거 agent session을 검색·질의할 수 있게 한 것도 같은 방향이다. 한마디로, **기억이 채팅 로그가 아니라 작업 단위로 재구성**되고 있다.

## 2) 실전에서는 "세션 브리프"부터 써야 한다

세션 중심 운영의 핵심은 시작할 때 이미 끝을 생각하는 것이다. 나는 항상 세션마다 아래 4가지를 먼저 적어둔다.

```md
### Session Brief
Goal: 무엇을 바꾸는가
Branch: 어떤 브랜치에서 작업하는가
Constraints: 건드리면 안 되는 것
Done when: 완료 기준은 무엇인가
```

이렇게 적어두면 에이전트에게도, 나중에 다시 보는 사람에게도 좋다. 특히 세션이 길어질수록 "왜 이 작업을 시작했는지"가 흐려지는데, 브리프가 그 기억을 붙잡아 준다.

완료 후에는 결과를 짧은 구조로 남기면 된다.

```json
{
  "goal": "login flow fix",
  "branch": "fix/login-state",
  "changed_files": ["app/auth.ts", "app/login.test.ts"],
  "open_risks": ["legacy cookie path"],
  "next_prompt": "add regression test for expired token"
}
```

이 JSON은 거창한 관찰 시스템이 아니다. 다음 세션을 빠르게 재개하기 위한 최소한의 기억장치다.

## 3) `/chronicle`는 요약이 아니라 재시작 버튼이다

`/chronicle`의 가치는 "예쁘게 정리된 리포트"에 있지 않다. 진짜 가치는 **다음 작업을 다시 시작할 수 있게 만든다**는 점에 있다. GitHub 문서는 Copilot CLI 세션 데이터가 검색 가능한 기록이 되고, 과거 세션을 자연어로 조회해 스탠드업 리포트나 개인화된 팁을 만들고, 이전 작업을 재개할 수 있다고 설명한다.

그래서 나는 세션 종료 시 항상 세 가지를 남긴다.

- 무엇을 바꿨는가
- 아직 남은 리스크는 무엇인가
- 다음 세션의 첫 프롬프트는 무엇인가

이 세 가지만 있으면 `/chronicle` 같은 검색형 기억이 훨씬 잘 작동한다. 반대로 이 정보가 없으면, 세션 기록이 있어도 결국 다시 처음부터 읽어야 한다.

## 4) 주의할 점: 세션이 길어질수록 분리해야 한다

세션 기반 워크플로우의 함정은 한 세션에 너무 많은 목적을 넣는 것이다. 그러면 브랜치도 흐려지고, 회귀도 늘고, 나중에 요약을 읽어도 "무엇이 결정사항이었는지"가 사라진다.

그래서 기준은 단순하다.

- 하나의 결과물에는 하나의 세션
- 하나의 세션에는 하나의 브랜치
- 하나의 브랜치에는 하나의 검증 기준

이렇게만 지켜도 에이전트는 훨씬 예측 가능해진다. 결국 중요한 건 모델이 아니라 **세션을 어떻게 설계하느냐**다.

## 결론: 이제는 프롬프트보다 세션이다

- GitHub Copilot 앱은 에이전트 작업의 데스크톱 홈으로 자리 잡고 있다.
- `/chronicle`와 Copilot Chat은 과거 세션을 다시 꺼내 쓰는 흐름을 만든다.
- 실전 운영의 핵심은 세션 브리프, 브랜치 분리, 종료 시 요약이다.
- 에이전트 품질은 답변 길이가 아니라 **재개 가능성**으로 판단해야 한다.

오늘 바로 할 일은 간단하다. 다음 에이전트 작업부터는 채팅창에 바로 요청하지 말고, 세션 브리프를 먼저 4줄로 적어보자. 그 순간부터 작업은 대화가 아니라 **운영 가능한 기록**이 된다.
