# 읽기·쓰기 MCP와 x402 결제를 분리해야 하는 이유

AI 에이전트가 실제 업무 시스템에 들어오면, 가장 먼저 부딪히는 문제는 모델 성능이 아닙니다. **누가 들어왔는지**, **무엇을 할 수 있는지**, **무엇에 돈을 내는지**가 한 덩어리로 엉켜 버리는 점입니다. 특히 Quote-to-Cash(QTC)처럼 가격, 견적, 계약, 청구, 리포팅이 이어지는 영역에서는 이 혼합이 더 치명적입니다.

최근 Turnstile이 공개한 흐름은 이 문제를 잘 보여 줍니다. 공식 제품 페이지는 Turnstile MCP를 “AI agents에게 Quote-to-Cash 시스템 오브 레코드에 대한 접근을 주는 secure interface”로 설명하면서, **read and write** 접근을 전면에 내세웁니다. 즉, 에이전트는 단순 질의만 하는 챗봇이 아니라, 실제로 QTC 데이터를 읽고 쓰는 행위 주체가 됩니다. 이때 필요한 설계는 분명합니다. **MCP는 맥락과 권한을, x402는 결제를** 맡아야 합니다.

## 1. MCP는 접근 제어, x402는 정산입니다

MCP authorization 스펙은 HTTP 기반 전송에서의 권한 부여를 다룹니다. 문서상으로 MCP의 authorization은 **옵션**이지만, HTTP 기반 transport를 쓰는 구현은 이 스펙을 따르는 것이 권장됩니다. 또한 스펙은 OAuth 2.1, RFC8414, RFC7591 같은 표준 위에 놓여 있습니다. 이 말은 곧, MCP가 “도구를 쓰는 문”이라면 OAuth는 “그 문을 열 수 있는 열쇠”라는 뜻입니다.

x402는 다른 문제를 다룹니다. x402 공식 문서는 이를 HTTP 위에서 동작하는 open payment standard라고 설명합니다. 핵심은 간단합니다. 서버가 먼저 `402 Payment Required`를 돌려주고, 클라이언트는 결제 정보를 붙여 다시 요청합니다. 계정, 세션, 복잡한 credential management 없이도 기계가 기계에게 결제할 수 있게 만드는 것이 목적입니다.

이 둘을 섞지 않는 이유는 명확합니다.

- **MCP/OAuth**는 “이 에이전트가 무엇을 할 수 있는가”를 정합니다.
- **x402**는 “이 요청이 유료 리소스에 해당하는가”를 정합니다.
- **QTC 시스템**은 그 둘을 바탕으로 실제 업무를 수행합니다.

## 2. 읽기와 쓰기를 먼저 분리하십시오

QTC를 에이전트에 열 때 가장 흔한 실수는 “검색”과 “수정”을 같은 도구로 묶는 것입니다. 그러나 실제 운영에서는 읽기와 쓰기의 위험도가 완전히 다릅니다.

예를 들어 다음처럼 나누는 편이 낫습니다.

```yaml
tools:
  qtc.quote.read:
    scope: ["qtc:read"]
    payment: "none"
  qtc.subscription.write:
    scope: ["qtc:write"]
    payment: "none"
  qtc.premium.report:
    scope: ["qtc:read"]
    payment: "x402"
```

이렇게 해 두면 운영이 훨씬 단순해집니다. 읽기 도구는 넓게 열되, 쓰기 도구는 scope를 좁게 잡고, 유료 리포트나 고비용 분석만 x402로 분리하면 됩니다. 에이전트에게 “아무거나 다 할 수 있는 하나의 슈퍼 권한”을 주는 순간, 감사 로그와 장애 분석은 급격히 어려워집니다.

## 3. 실전 흐름은 401, 403, 402로 읽으십시오

상태 코드를 분리하면 디버깅이 쉬워집니다.

```python
def route_request(auth_ok: bool, scope_ok: bool, paid_ok: bool):
    if not auth_ok:
        return 401  # 인증 실패
    if not scope_ok:
        return 403  # 권한 부족
    if not paid_ok:
        return 402  # 결제 필요
    return 200
```

이 작은 분기만으로도 운영팀은 로그를 훨씬 빠르게 읽을 수 있습니다. 401이 많으면 인증 흐름을 고쳐야 하고, 403이 많으면 scope 설계를 다시 봐야 하며, 402가 많으면 과금 단위나 가격 장벽이 너무 높다는 뜻일 수 있습니다.

x402의 MCP 서버 가이드는 이 흐름을 더 구체적으로 보여 줍니다. MCP 서버가 외부 유료 API를 호출하다가 `402`와 `PAYMENT-REQUIRED`를 받으면, 결제 요구를 해석하고, 적절한 scheme으로 서명한 뒤, `PAYMENT-SIGNATURE`를 붙여 재시도합니다. 성공하면 paid data를 그대로 클라이언트에 돌려줍니다. 즉, 결제는 사용자의 습관이 아니라 **프로토콜 레벨의 재시도 로직**이 됩니다.

## 4. Turnstile가 보여 준 것은 “도구”가 아니라 “원장”입니다

Turnstile의 제품 소개를 자세히 보면 중요한 메시지가 하나 더 있습니다. 이 회사는 단순히 에이전트가 접근할 수 있는 API를 만든 것이 아니라, **pricing, quotes, customer agreements, billing, reporting을 하나의 Quote-to-Cash system of record로 묶는 구조**를 제안합니다. MCP는 그 원장에 접근하는 secure interface이고, 그 위에서 AI agents는 질문을 던지고, 추천을 받고, 실제 작업까지 수행합니다.

이 관점이 중요한 이유는, 에이전트가 늘어날수록 “챗봇이 무엇을 아느냐”보다 “시스템의 진실이 어디에 있느냐”가 더 중요해지기 때문입니다. 에이전트가 읽는 데이터와 사람이 보는 데이터가 다르면, 결국 승인과 정산이 꼬입니다. 그래서 QTC는 별도의 AI 레이어가 아니라, **읽고 쓰는 동일한 원장**이어야 합니다.

## 5. 구현할 때 꼭 지켜야 할 체크리스트

x402 문서는 결제 요청을 만들기 전에 몇 가지 정책 검사를 권합니다. 예를 들어 최대 금액, 통화, 예상 네트워크와 scheme, 대상 resource server, 그리고 사용자·에이전트·세션별 지출 한도를 먼저 확인하라는 식입니다. 이 순서를 지키면 에이전트가 결제를 남발하는 일을 줄일 수 있습니다.

실무 체크리스트는 다음 정도면 충분합니다.

- 읽기 도구와 쓰기 도구를 먼저 분리하십시오.
- OAuth scope는 기능 단위로, x402 결제는 상품 단위로 설계하십시오.
- 결제 실패와 권한 실패를 같은 에러로 처리하지 마십시오.
- 모든 write 작업에는 감사 로그를 남기십시오.
- x402는 메터링된 고비용 리소스에만 붙이십시오.

## 결론

에이전트 상거래는 결국 세 가지를 분리하는 일입니다.

- **MCP**는 무엇을 읽고 쓸 수 있는지 정합니다.
- **OAuth**는 누가 그 일을 할 수 있는지 정합니다.
- **x402**는 무엇에 돈을 내야 하는지 정합니다.

Turnstile의 read-and-write MCP는 이 방향이 이미 제품화 단계에 들어왔다는 신호입니다. 다음 단계의 설계는 단순합니다. **읽기 도구 하나, 쓰기 도구 하나, 유료 도구 하나**를 골라서 각각 다른 정책으로 분리해 보십시오. 그 한 번의 분리만으로도 에이전트 시스템은 훨씬 덜 위험하고, 훨씬 더 운영 가능해집니다.
